
# GSS-OS

**一种面向端侧小模型的极简 AI 操作系统控制接口层。**

GSS-OS（General Skills Standard - Operating System）不是传统意义上的"给大模型加工具"的框架。  
它试图把 AI 桌面控制从**重型截图导航 + 复杂 JSON / schema 输出**，重写为：

- **原子文本指令**
- **Grid Domain 格子域坐标**
- **动态按需分库**
- **HOW-code 工具自举**
- **Hard Freeze 硬冻结安全审查**

> **核心思想：不是让模型更大，而是让任务表示更适合小模型。**

---

## 为什么做 GSS-OS

当前很多 GUI Agent / Computer Use 系统可以工作，但通常伴随几个工程问题：

- 依赖整屏截图和重型视觉推理
- 要求模型稳定输出复杂结构化 schema / JSON
- 常驻 prompt 和工具说明膨胀严重
- 小模型很难胜任
- 本地化与断网部署能力弱
- 工具扩展缺乏强审查边界

GSS-OS 的目标不是否定这些系统，而是提供一种**更轻、更快、更适合本地小模型**的替代路径。

---

## 一句话理解

传统 GUI Agent 常常让模型：

> **看整张屏幕 → 理解界面 → 规划动作 → 输出复杂调用**

GSS-OS 尽量把问题改写成：

> **读取最小任务状态 → 输出短文本原子命令 → 本地解析执行**

---

## 最小示例

任务：

> 打开记事本，等待一秒，输入"人类，你好！我是一个AI操作系统"，然后全选这些文字。

模型输出：

```text
o notepad
sl 1
t 人类，你好！我是一个AI操作系统
hk Ctrl+A
```

执行器将其解析为真实桌面动作并本地执行。

## 四个核心模块

### 1. 16 条原子指令

GSS-OS 将桌面控制压缩为固定的 16 条原子指令，可视为一种 **AI Assembly Language**：

```text
MOUSE: c <g> | dc <g> | rc <g> | sc <dir> <n> | dr <g1> <g2>
KEY  : hk <combo> | t <text>
PROC : o <path> | k <name> | w <title>
FLOW : sl <sec>
KNOW : ld <lib>
EVO  : hc <name> | mk_lib <lib> | mv <tool> <lib>
COMM : msg <text>
```

原则：

- 原子化
- 极短文本
- 可预测输出
- 快捷键优先
- 小模型友好

---

### 2. Grid Domain（格子域）

GSS-OS 不要求模型始终理解整张屏幕像素。  
它将屏幕离散为 1–8 层自适应网格，把 GUI 定位改写为**文本坐标选择问题**。

例如：

```text
c C4
dr A1 D8
```

这让不具备重型视觉能力的小模型也能进行桌面定位。

---

### 3. 动态分库（Dynamic Library）

系统不把所有工具和知识一次性注入上下文，而是按需加载：

```text
ld lib_web
ld lib_sys
```

这对应 GSS 的最小使用集原则：  
**当前任务只激活当前需要的能力。**

---

### 4. HOW-code + Hard Freeze

当现有工具不足时，执行者模型可发出：

```text
hc img_zip
```

这不会直接执行代码，而会触发：

1. 工具规格生成
2. 创造者模型生成代码
3. **Hard Freeze 硬冻结**
4. 人类审查
5. 批准后再注册为未来能力

> 工具生成不是普通动作调用，而是能力边界修改。  
> 因此，未经审查不得入库，不得执行。

---

## 为什么这条路线对小模型更友好

GSS-OS 的核心不是"让模型更强"，而是**降低操作表示复杂度**。

与重型 GUI Agent 相比，它尽量避免让模型同时承担：

- 整屏视觉理解
- 长上下文处理
- 复杂 schema 生成
- 高维动作回归

而是尽量只要求模型输出短文本命令序列。

---

## 当前经验数据

在已定义桌面控制任务中，我们的经验结果为：

- **GSS-OS 单次行动消耗**：约 **120–180 tokens**
- **传统重型框架**：常在 **3000–7000+ tokens**

- **GSS-OS 首条动作延迟**：约 **0.3–1.0 秒（本地 7B）**
- **传统重型方案**：常在 **数秒到分钟级**

> 这些结果目前主要来自早期工程原型与已定义任务，不代表所有场景的通用结论。

---

## 安全边界

GSS-OS 的安全模型依赖：

- 白名单解析器
- 原子指令集
- 参数校验
- 动态库加载控制
- Hard Freeze 人工审查
- 不让未审查代码直接进入执行面

最重要规则：

> **在用户作出明确选择前，系统必须停滞。**

---

## 边界说明

GSS-OS 并不声称替代所有 Agent 框架，也不适合所有任务。

它当前最适合：

- 操作系统桌面控制
- 本地小模型运行
- 低 token / 低延迟环境
- 离线 / 真断网 / 隐私敏感场景
- 可审查执行场景

它当前不主打：

- 复杂长结构化 API 编排
- 高维图像语义理解
- 通用视觉问答
- 完整替代 DOM / Accessibility Tree 系统

对于部分信息提取任务，GSS-OS 仍可能使用截图作为**最后补充机制**。  
但其原则是：**导航尽量不用截图。**
